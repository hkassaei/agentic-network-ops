## Post-Run Analysis

### Summary of what went wrong

Final score **30 %**. The agent correctly observed that `mongo` was exited and correctly named the `infrastructure` layer as RED, but then its Synthesis **explicitly dismissed the mongo outage as unrelated to the observed IMS registration failures**, attributing the problem instead to "an undetermined fault within the HSS (pyhss)."

The failure happened at three distinct levels, each interesting in its own right:

1. **NA hallucinated a mongo → HSS dependency** that the ontology does not assert.
2. **Investigator h1 "disproved" the correct hypothesis** using evidence that — taken at face value — was actually consistent with reality.
3. **The ontology is missing the edge that matters**: mongo → UDR → PCF → Rx AAR → P-CSCF.

Walking each of these yielded a broader, architectural finding about how we built the ontology — captured at the end of this analysis.

### 1. The NA's hallucinated mongo → HSS link

NA's Phase 3 summary reads:

> …the root cause is very likely an outage of the `mongo` database preventing the HSS from accessing user data.

But the ontology says the opposite. Per `network_ontology/data/components.yaml`:

- `mongo`: *"5G subscriber data store (UDR backend). Stores subscriber profiles, session state, and policy data."*
- `mysql`: *"IMS subscriber data store. **Backend for PyHSS and Kamailio CSCFs.**"*
- `pyhss`: *"Home Subscriber Server. Handles Cx Diameter queries from I/S-CSCF."*

There is no authored edge from `mongo` to `pyhss`. The NFs that actually depend on mongo per the ontology are `udr`, `udm`, and `pcf` — all of which NA correctly named in its own CORE evidence block:

> **CORE evidence:**
> - get_network_topology: PCF and UDR have inactive links to MongoDB

But then NA's summary pivoted to HSS anyway — an NF that **does not appear in NA's own gathered evidence** for this layer. Token accounting shows `NetworkAnalystAgent: 4 tool calls`, accounting for `get_network_status`, two `get_network_topology` calls, and one `get_nf_metrics`. There is no call to `get_causal_chain_for_component` or the `OntologyConsultationAgent` — the two tools that would have returned the authoritative "what happens when mongo dies" story. NA reached past its own evidence and its tools to borrow a generic LLM prior about HSS being MongoDB-backed in other deployments.

### 2. Investigator h1's "disproof" was technically correct but situationally wrong

h1 cited `query_subscriber("001011234567891")` returning `core_5g: null` + `ims_subscriber: {...}` as decisive falsification:

> This proves that the HSS uses a different, operational database and is not affected by the MongoDB outage.

That claim is *ontology-consistent* — pyhss does use mysql, not mongo. But the framing inside h1's hypothesis ("mongo down → HSS unreachable") was never the right framing. The correct story has mongo affecting **PCF**, not HSS — so disproving mongo → HSS leaves the actual fault chain (mongo → PCF) unexamined. The Investigator faithfully falsified the wrong target; the whole branch was mis-framed upstream in NA.

### 3. The real causal chain — and the ontology gap that hides it

Combining `components.yaml`, `flows.yaml`, and the Kamailio implementation in `network/pcscf/route/mo.cfg:360-395` and `mt.cfg:67-83`, the actual chain from `mongo` death to `pcscf_sip_error_ratio = 1.00` is:

```
mongo dies
  → UDR blind (UDR's backend is mongo — ontology says so)
    → PCF can't fetch subscriber policy (PCF's data path is PCF → UDR → mongo)
      → P-CSCF ↔ PCF Rx AAR times out or returns error
        → mo.cfg:392-395: dlg_terminate("all", "Sorry no QoS available")
          → P-CSCF returns SIP error on the INVITE
            → pcscf_sip_error_ratio → 1.00, on EVERY call setup attempt
```

Two specific implementation facts make this chain inevitable:

- **Kamailio fires `Rx_AAR` inline with the 180/183/200 response handler**, not as a post-establishment hook. From `mo.cfg:360-373`:
  ```
  if (t_check_status("180|183|200") && has_body("application/sdp")) {
      xlog("L_DBG","Diameter: Orig authorizing media via Rx\n");
      if (Rx_AAR("MO_aar_reply","orig","",-1) == 0) {
          exit;
      }
  }
  ```
  So Rx AAR is a **gate on call setup**, not an afterthought. If AAR fails, the dialog is torn down with `"Sorry no QoS available"` before the 200 OK reaches the caller.
- **Every VoNR INVITE hits this gate.** The observation-traffic agent places ~one call every ~20 s during the 2-minute window, and every one of them needs Rx AAR to succeed. When mongo is down, every one fails. That's why the ratio is 100 %, not some fraction.

**Neither the ontology nor NA produced this chain.** `causal_chains.yaml`'s `subscriber_data_store_unavailable` entry (lines 577-608) names mongo's consequences as:

> cascading:
>   - "UDR cannot query subscriber data, **SMF session creation fails**"
>   - "existing sessions may survive briefly…"

It traces mongo → UDR → SMF (new PDU-session failure). The UDR → PCF → Rx → P-CSCF branch — the one carrying 100 % of the observed SIP error load in this scenario — is **absent from the ontology entirely**. Even if NA had dutifully consulted `get_causal_chain_for_component("mongo")`, it would have returned an incomplete story that didn't explain the P-CSCF symptom at all.

The 30 % UAR-timeout rate at I-CSCF (9 timeouts / 30 attempts) is a separate smaller symptom. Per the ontology and the `query_subscriber` evidence, pyhss itself is healthy. The timeouts are most plausibly secondary effects (UEs failing 5G re-attach at the AUSF→UDM→UDR→mongo path, retrying stale REGISTERs) or load/timing artifacts. They are not the driver of the 100 % SIP error ratio.

### 4. `flows.yaml` also has the wrong Rx AAR step ordering

As a side finding during the causal-chain investigation, I verified that `network_ontology/data/flows.yaml`'s `vonr_call_setup` flow places Rx AAR as **step 10, after step 9 "200 OK + ACK — Call established."** In the actual implementation, Rx AAR fires **inside step 9's response handling**, gated on `180|183|200` with SDP body, and a failed AAR tears the dialog down with `dlg_terminate("all", "Sorry no QoS available")` — meaning the dialog is never "established" if AAR fails.

This positioning error propagates into the agent's mental model: a pipeline that reads flows.yaml as ground truth comes away thinking "Rx AAR is an optional post-establishment QoS enhancement," which leads to the specific wrong conclusion that mongo-down-via-PCF would degrade dedicated QoS but not cause SIP errors. That is exactly the inference NA did *not* make — it never reached "mongo → PCF → Rx → P-CSCF" at all — but the flow as written would not have guided it there even if it had consulted the flow.

## The broader architectural finding

**Ontology flows must be anchored in the actual implementation, not in formal 3GPP specifications or LLM prior knowledge.** This run makes the cost concrete.

### What the ontology currently is

Today's `flows.yaml`, `causal_chains.yaml`, and the `meaning.*` / `failure_modes` blocks in `metrics.yaml` appear to be **textbook-based 3GPP skeletons populated with stack-specific observations in the obvious fields** (metric names, container names, numeric ranges). The textbook skeleton is why:

- `vonr_call_setup` step 10 places Rx AAR after "Call established" — canonical sequence-diagram abstraction, not what Kamailio actually does on the SIP response path.
- Step 10's `failure_modes` say *"PCF unreachable → no dedicated QoS, voice uses default bearer"* — the idealized 3GPP fallback behavior, not the implementation, which does `dlg_terminate("all", "Sorry no QoS available")` on AAR failure.
- `subscriber_data_store_unavailable`'s cascading list names `SMF` only — the textbook-obvious consequence — and misses `PCF/Rx/P-CSCF`, which is the consequence *this specific Kamailio + Open5GS setup* actually exhibits.

These are all places where what's written matches the textbook *shape* of IMS/5G but diverges from what the code does. A human reading them would not be misled — they'd recognize the abstraction level. **An LLM agent treating them as ground truth, as ours does, has no way to know where the textbook stops and the implementation begins.**

### Why this is worse than having no ontology

- **No ontology** = NA falls back entirely to its own priors. Hallucination risk is high, but at least everything in the reasoning chain is explicitly LLM-generated and can be corrected by prompt rules.
- **Textbook ontology treated as ground truth** = NA has a seemingly-authoritative reference that is *subtly wrong*. The agent doesn't feel the need to verify because the ontology says so. Errors become harder to spot because they're cloaked in structured, confident-looking YAML.

In this run, the ontology's silence on mongo→PCF→Rx and its mis-ordered Rx AAR in flows.yaml combined to give NA no authored path to the correct answer, while simultaneously giving it textbook-shaped framings that it could pattern-match against its LLM priors to produce the HSS hallucination.

### What "implementation-anchored" actually means

Three concrete shifts, in order of leverage:

1. **Every flow step that represents code execution must carry an `implementation_ref` field** pointing at the file / lines / symbol in this repo that enacts it (e.g. `{file: "pcscf/route/mo.cfg", lines: "360-395", symbol: "Rx_AAR(orig)"}`). Authors can't put down a step without pointing at the code that does it. Readers (human or agent) can verify any step in one click.
2. **`failure_modes` for implementation-backed steps must enumerate the actual error branches in the code**, not imagined-3GPP fallbacks. Rx AAR failure's real branch is `dlg_terminate("all", "Sorry no QoS available")` → SIP 5xx upstream → dialog never reaches ACK state. That's what should be in `failure_modes`. The textbook "voice uses default bearer" fallback is not what our code does.
3. **`causal_chains.yaml` entries for infrastructure failures must trace every downstream consumer named in `components.yaml`**. If `components.yaml` says UDR → mongo *and* PCF → UDR *and* PCF ↔ P-CSCF on Rx, then `subscriber_data_store_unavailable`'s `cascading` block must spell out the PCF → Rx → P-CSCF branch, not just the SMF one. Authoring the edge once is cheap; letting the agent miss it repeatedly is expensive.

### Immediate impact on how we proceed

The mongodb_gone 30 % score is not fixable at the prompt layer. No amount of "be more careful" instructions to NA will teach it that our Kamailio Rx AAR tears down dialogs on PCF failure. That knowledge has to be authored into the ontology — from the code, not from 3GPP TS 23.228.

This finding also retrospectively explains several smaller reasoning defects seen across prior runs: whenever the agent has confidently produced a slightly-wrong causal narrative (e.g., the P-CSCF → PCF latency misread in an earlier run, or the call_quality invented signaling fault), the pattern has been "the ontology was silent, so the agent filled in from priors." The fix is not prompt-side restraint; it's ontology-side authoring discipline.
