**Status:** Accepted — implementation pending

## Context

The v6 Investigator has two log-based tools in its toolset:

- **`read_container_logs(container, grep, since_seconds)`** — runs `docker logs <container>` and optionally filters by a caller-supplied regex.
- **`search_logs(containers, pattern)`** — multi-container variant of the same.

These were included to give the Investigator a way to verify application-level error conditions described in hypotheses. In practice the Instruction Generator (IG) and the Investigator itself construct probes of the form:

```
read_container_logs(container="pcf", grep="mongo|database|connection refused")
```

…where the grep pattern is generated by the LLM from its prior knowledge of what the component "probably logs" when failing. The hypothesis is confirmed or disproved based on whether the grep returns matches.

This approach has a specific, recurring, and high-impact failure mode.

---

## The Failure Pattern

Across three distinct `MongoDB Gone` episodes (and likely in other scenarios we haven't catalogued as carefully), the Investigator has disproved the **correct** hypothesis by over-weighting empty log-search results. The pattern is:

1. NA correctly identifies the causal chain (e.g. `mongo → UDR → PCF → P-CSCF`).
2. IG writes a plan that includes one `get_network_status` / `get_nf_metrics` probe (which confirms the hypothesis) plus one or more `search_logs` probes with narrow grep patterns.
3. The narrow grep patterns don't match the component's actual log vocabulary (Open5GS, Kamailio, and PyHSS all use different error terminology, some of which depends on compile flags like `WITH_RX` / `WITH_N5`).
4. Investigator classifies the empty log-search result as `CONTRADICTS`.
5. Investigator applies vote-counting: 2 × CONTRADICTS vs 1 × CONSISTENT ⇒ verdict `DISPROVEN`.
6. Synthesis trusts the DISPROVEN verdict, produces a diagnosis of *"undetermined fault"* or an alternative-suspect hallucination.

The common element: **absence of log matches is treated as strong evidence of the hypothesis being wrong**, when in reality it is weak or zero evidence — the grep pattern simply didn't intersect the component's actual log output.

### Why this keeps happening

Several compounding factors, each valid in isolation but structurally incompatible with the Investigator's current falsification contract:

- **Log vocabularies vary by component.** Open5GS writes its own format; Kamailio uses `xlog` with operator-chosen tags; PyHSS uses Python logging. No canonical error vocabulary exists across the stack.
- **Log vocabularies vary by compile flag and configuration.** Kamailio's `WITH_RX` being compiled out means the string "Rx" never appears in P-CSCF logs — regardless of whether the hypothesis is true or false. The LLM has no way to know this at probe-authoring time.
- **LLMs fabricate plausible-looking grep patterns.** The IG writes patterns like `"mongo|database|connection refused"` or `"Rx|policy|PCF"` based on what "failing" components "usually log" in general 3GPP/DB literature. Those patterns frequently do not match our specific stack.
- **Absence-of-match is not absence-of-failure.** A component may be failing silently, logging at a different level (DEBUG vs ERROR), logging to a different sink (stdout vs file), or logging at a different moment (setup vs steady-state). None of these mean the hypothesis is wrong.
- **LLMs default to vote-counting.** With 1 CONSISTENT and 2 CONTRADICTS, the Investigator LLM consistently concludes DISPROVEN, even when explicit prompt rules tell it to weight evidence strength over probe count. We've tried three revisions of the evidence-weighting rule; the vote-counting default wins in practice.

The net result: logs are a **structurally weak signal** in our current probe framework, yet they are being treated as **structurally authoritative** by the Investigator. That is the worst possible combination.

---

## Decision

Remove `read_container_logs` and `search_logs` from the agent-facing toolsets for the v6 pipeline. Specifically:

1. Remove both tools from the `Investigator` subagent's `tools=[...]` list in `agentic_ops_v6/subagents/investigator.py`.
2. Remove both tools from any Instruction Generator plan catalog entry (the IG prompt's "Investigator's available tools" table) so the IG stops writing log-based probes.
3. **Retain the backend implementations** in `agentic_ops_common/tools/log_search.py` and `_common.py`. They are still valid as lower-level primitives; we are removing them from *agent-facing* exposure, not deleting them.
4. Retain `OntologyConsultationAgent.interpret_log_message` if and when it is used — that tool takes a pre-observed log line and returns semantic interpretation; it is NOT the same thing as agent-authored grep-based log retrieval.

The Investigator continues to have the structured-metric path (`get_nf_metrics`, `get_dp_quality_gauges`), the status path (`get_network_status`, `check_process_listeners`), the connectivity path (`measure_rtt`), the configuration-reading path (`read_running_config`, `read_env_config`), the Kamailio-state path (`run_kamcmd`), and the ontology paths (causal chains, flows, subscriber lookup). These are all structured observations that don't rely on grep patterns matching unknown vocabularies.

### What we are giving up

- **Legitimate log-based evidence** exists. When a component crashes with an uncaught exception, its logs often contain a stack trace that is direct evidence of the fault. When a Diameter peer rejects a request, the responder often logs a specific error code. Removing the tool removes access to this signal.
- **Human operators read logs during troubleshooting.** The Investigator is modeled on a NOC engineer's workflow, and that workflow does include log inspection. Removing logs makes the Investigator less lifelike, which may reduce its ability to surface failure modes that are genuinely log-visible.

We accept these costs in exchange for eliminating a documented source of recurring mis-diagnoses. The cost of a confident-but-wrong answer (score 30% / 42% with a fabricated alternative root cause) is, in our judgement, worse than the cost of a less-confident correct answer that relies on structured metrics alone.

### Scope

- The removal applies to the **v6 pipeline**. v5 and prior pipelines are unaffected; their log tools remain because they are deprecated but not broken.
- Other `agentic_ops_common` consumers (scripts, tests, ad-hoc tools) are unaffected — the backend is retained.

---

## Consequences

### Positive

- **The mongodb_gone failure mode is closed** at the evidence layer. The IG can no longer write narrow-grep log-search probes because the tool is not in the Investigator's list; the Investigator can no longer be fooled by empty log searches because it cannot run them.
- **One less class of prompt rule to maintain.** We no longer need the tortured "evidence weighting" language trying to coach the LLM on how to interpret absent log matches. The prompt surface shrinks.
- **Forces the IG toward stronger probe shapes.** Without log-search as a default cheap probe, the IG has to reach for `get_nf_metrics` and `get_network_status` — both of which are direct observations that don't depend on vocabulary matching.
- **Parallels the prior `query_prometheus` removal** for consistent architectural reasoning: agent-authored free-form query strings are a consistent hallucination surface. Log-search was the second such surface; removing it completes the pattern.

### Negative / risk

- **Some failure modes are genuinely log-visible** and we lose direct observation of them. Examples: container OOM-kill messages, uncaught exceptions with stack traces, specific protocol error codes that are logged but not metric'd. For those cases the Investigator will have to infer from secondary evidence (status, metrics, configuration) which may be less direct.
- **A future scenario may have its correct answer visible only in logs.** We do not currently know of such a scenario among the 11 chaos tests, but the space is larger than our test set. If this happens we will see it as a failure and revisit.
- **The Ontology Consultation Agent retains `interpret_log_message`,** which takes a pre-captured log line as input. It is not a substitute for grep-based retrieval, and we are not claiming it is. Agents will no longer have a way to obtain log lines in the first place — so the interpretation tool becomes useful only for logs surfaced through other channels (e.g. captured by the chaos framework's baseline snapshots). Worth noting for future work.

### Alternatives considered

1. **Add a "log-pattern validation" step to the IG prompt** (*"before proposing a grep pattern, verify it against a known-pattern catalog"*). Rejected: requires us to maintain a catalog of known log patterns per component, per compile flag, per version. High authoring cost; catalog staleness guaranteed; still prone to LLM pattern-matching errors at authoring time.
2. **Add a "log grep pattern safety rating" heuristic** at runtime (the tool could return a rating alongside results). Rejected: doesn't address the structural issue. A "safe" grep pattern can still return no matches when the hypothesis is true.
3. **Switch to structured log retrieval via a log indexer** (e.g., Loki, Elasticsearch, or a custom parser that extracts structured fields per component). Recommended as future work but out of scope for this decision. Would require: a per-component parser catalog, a storage/indexing service, and an agent-facing query tool that speaks the structured schema rather than regex. This is the "right way" but needs dedicated design.
4. **Keep the tools but cap their evidence weight in the verdict** (enforce mechanically in the orchestrator: `search_logs → no matches` never produces `CONTRADICTS`, only `AMBIGUOUS`). Rejected for now: the Investigator's verdict is free-text + structured output from an LLM; the orchestrator does not currently parse probe-level verdicts. Adding this enforcement would be a larger code change than removing the tools. Revisit if the tools return.
5. **Stronger prompt engineering** (we have tried three revisions of the evidence-weighting rule). Rejected as a sole fix: prompt-layer mitigations have not held. The structural mismatch between "agent-authored grep pattern" and "absence-of-match as evidence" is the issue.

## Implementation manifest

### Files to change

- `agentic_ops_v6/subagents/investigator.py` — remove `tools.read_container_logs` and `tools.search_logs` from the `tools=[...]` list.
- `agentic_ops_v6/prompts/investigator.md` — remove both from the "Tool constraint" section's allowed-tools list. Remove / tighten any example that relies on log-based probes.
- `agentic_ops_v6/prompts/instruction_generator.md` — remove both from the "Investigator's available tools" table in the IG prompt. Remove examples that reference log-grep probes. Remove or re-anchor any existing rule text that implies logs are a probe option.
- `agentic_ops_common/tools/__init__.py` — optional: move `read_container_logs` and `search_logs` out of `__all__` (keep the imports but signal they are not for agent use), mirroring what we did for `query_prometheus`.

### Files explicitly NOT changed

- `agentic_ops_common/tools/log_search.py` — backend implementation stays. Accessible via the module directly, just not re-exported for agent wiring.
- `agentic_ops_v5/*` and earlier — log tools remain. This decision applies to v6 only.

### Verification

After the change:
- Re-run the `MongoDB Gone` scenario. The Investigator should no longer produce a DISPROVEN verdict on the correct hypothesis, because the mechanism that was producing it (empty log searches counted as CONTRADICTS) no longer has a path to trigger.
- Run the full 11-scenario batch. Watch for scenarios that previously scored well but may now score worse because their correct answer relied on a log observation (no scenarios in our current test set are expected to fall into this category, but this is the risk we are accepting).

## Follow-on work

This decision is deliberately narrow: remove the weak signal, move on. The longer-term work is:

- **Structured log extraction.** Build a per-component log-parser catalog that emits structured events (e.g., `{event: "mongo_connection_failed", container: "udr", timestamp: ..., detail: ...}`) and make *those* available to the Investigator. The Investigator would then query structured log events with field filters, not grep strings. Far more reliable; the absence-of-match problem goes away because events only exist when the parser recognized them.
- **Log-anomaly detection.** Similar in spirit to the metric anomaly screener: a background process that identifies unusual log lines and surfaces them as typed events in the event store. Agents consume the events, not the raw text.
- **RAG over log archives.** Once there is historical log data, semantic retrieval over past incidents provides stronger signal than live grep. The Episode-RAG track (mentioned in `falsifier_investigator_and_rag.md`) subsumes this.

Each of these paths produces the same thing from the agent's perspective: **structured log observations rather than agent-authored grep queries.** Until at least one of those is available, the Investigator should not have access to logs.

---

## Reversal criteria

This ADR can be reversed if:
- A per-component log-parser catalog (or equivalent structured-log-event stream) is built, exposed via an agent-facing tool, and demonstrated to produce reliable CONTRADICTS / CONSISTENT verdicts on scenarios where the current grep-based tool fails.
- A specific scenario emerges in which the correct diagnosis is genuinely only surfaceable via log inspection and no structured-metric path exists — and a prompt-layer mitigation for the narrow-grep failure has been independently validated on simulated cases.

In either case, the reversal should be a new ADR that supersedes this one.
