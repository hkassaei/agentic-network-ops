# ADR: Agentic Ops v5 — 6-Phase Agent Pipeline

**Date:** 2026-04-01
**Status:** Accepted
**Supersedes:** `agentic_ops_v5_plan.md`, `v5_implementation_plan.md`

---

## Decision

Replace the v5 4-phase pipeline (Triage → OntologyAnalysis → Investigator → Synthesis) with a 6-phase pipeline where every phase is an agent (BaseAgent or LlmAgent) and the orchestrator is pure workflow plumbing.

## Context

The original v5 design (documented in `agentic_ops_v5_plan.md`) proposed a "deterministic backbone" where Phase 2 was a Python function that ran ontology queries in code. This design had three problems:

1. **Monolithic Phase 2:** Pattern matching, anomaly detection, and instruction generation were combined in a single `consultation.py` file. When the pattern matcher found nothing, the system fell back to a generic "investigate everything" mandate with no intermediate analysis.

2. **Missing Tier 2:** There was a gap between exact ontology match (Tier 1) and blind bottom-up investigation (Tier 3). When the ontology couldn't find an exact signature, the investigator received no guidance about which layer or component to focus on.

3. **Instruction quality:** The investigator's mandate was generated by Python string templates, not by an LLM that could synthesize context intelligently. Template-based mandates were rigid and couldn't adapt to novel combinations of symptoms.

## Pipeline Architecture

```
Phase 1: TriageAgent (LLM)              → state["triage"]
Phase 2: PatternMatcherAgent (BaseAgent) → state["pattern_match"]
Phase 3: AnomalyDetectorAgent (LLM)     → state["anomaly_analysis"]      (optional)
Phase 4: InstructionGeneratorAgent (LLM) → state["investigation_instruction"]
Phase 5: InvestigatorAgent (LLM)         → state["investigation"]
Phase 6: SynthesisAgent (LLM)            → state["diagnosis"]
```

### Phase 1: TriageAgent
- **Type:** LlmAgent (gemini-2.5-flash)
- **Tools:** get_network_topology, get_network_status, get_nf_metrics, read_env_config
- **Job:** Collect raw health data. No diagnosis, no reasoning — pure data collection.
- **Output:** `state["triage"]` — structured health report (topology, metrics, container status)

### Phase 2: PatternMatcherAgent
- **Type:** BaseAgent (deterministic, no LLM)
- **Tools:** None (calls OntologyClient directly in Python)
- **Job:** Collect structured observations from network tools, then query the ontology for matching failure signatures and baseline anomalies.
- **Output:** `state["pattern_match"]` — JSON with matched signatures, confidence, failure domain, baseline anomalies, raw observations
- **Key behavior:** If confidence is "very_high" or "high", Phase 3 is skipped.

### Phase 3: AnomalyDetectorAgent
- **Type:** LlmAgent (gemini-2.5-flash)
- **Tools:** check_stack_rules, compare_to_baseline, interpret_log_message, check_component_health, get_disambiguation, get_causal_chain, get_causal_chain_for_component
- **Job:** When no exact signature matches, analyze triage data using ontology tools to detect anomalies, group by network layer, identify suspect components and protocols.
- **Output:** `state["anomaly_analysis"]` — layer status (GREEN/RED), triggered stack rules, baseline deviations, suspect components
- **Key behavior:** Skipped when Phase 2 returns a high-confidence match. This is the Tier 2 that was missing from the original v5 design.
- **Constraint:** Does NOT diagnose — only organizes findings for the Instruction Generator.

### Phase 4: InstructionGeneratorAgent
- **Type:** LlmAgent (gemini-2.5-flash)
- **Tools:** None (pure text synthesis)
- **Job:** Synthesize triage data, pattern match results, and anomaly analysis into a crisp, focused instruction for the Investigator.
- **Output:** `state["investigation_instruction"]` — the investigator's mandate
- **Key behavior:** Produces different instructions depending on prior phases:
  - High-confidence match → "ESTABLISHED FACT: [diagnosis]. Verify it."
  - Anomalies detected → "Anomalies in [layer]. Investigate [components]."
  - Nothing found → "Full bottom-up investigation."

### Phase 5: InvestigatorAgent
- **Type:** LlmAgent (gemini-2.5-pro)
- **Tools:** All network diagnostic tools (measure_rtt, check_process_listeners, query_prometheus, get_nf_metrics, get_network_status, run_kamcmd, read_running_config, read_container_logs, search_logs, read_env_config, query_subscriber) + AgentTool(OntologyConsultationAgent)
- **Job:** Follow the investigation instruction, call tools to verify/disprove the hypothesis, and produce evidence-backed findings.
- **Output:** `state["investigation"]` — evidence chain with tool citations
- **Key behavior:**
  - Every claim must cite a tool output: `[EVIDENCE: tool_name("args") → "output"]`
  - Follows Hierarchy of Truth: Transport > Core > Application
  - Can call OntologyConsultationAgent (via AgentTool) for follow-up ontology queries mid-investigation

### Phase 6: SynthesisAgent
- **Type:** LlmAgent (gemini-2.5-pro)
- **Tools:** None
- **Job:** Fact-check investigator findings against prior context, apply Hierarchy of Truth, produce NOC-ready diagnosis.
- **Output:** `state["diagnosis"]` — final report with root cause, timeline, recommendation

## Orchestrator Design

The orchestrator (`orchestrator.py`) is pure workflow plumbing. It:
1. Runs each phase via `_run_phase()` with session isolation (no conversation history leaks between phases)
2. Checks `state["pattern_match"]["confidence"]` to decide whether to skip Phase 3
3. Catches phase failures gracefully (each phase has try/except with meaningful fallback)
4. Tracks token usage and timing per phase via `PhaseTrace`

The orchestrator contains no reasoning logic, no ontology queries, no template strings. Every decision is made by an agent.

## OntologyConsultationAgent (AgentTool)

The OntologyConsultationAgent is an LlmAgent (gemini-2.5-flash) with all ontology tools. It is wrapped as an `AgentTool` and given to the InvestigatorAgent. This allows the investigator to consult the ontology for follow-up queries during Phase 5 — for example, when it discovers a new log message or encounters an ambiguous situation not covered by the Phase 2/3 analysis.

## State Flow

```
Phase 1 writes: state["triage"]
Phase 2 writes: state["pattern_match"]
Phase 3 writes: state["anomaly_analysis"]     (or "Skipped" if Phase 2 high-confidence)
Phase 4 writes: state["investigation_instruction"]
Phase 5 writes: state["investigation"]
Phase 6 writes: state["diagnosis"]
```

Each phase sees all prior state keys via ADK prompt placeholders (`{triage}`, `{pattern_match}`, etc.).

## Design Principles

1. **Every phase is an agent.** The orchestrator contains no reasoning logic.
2. **Accuracy over speed.** 6 phases is more LLM round-trips, but each agent has a focused job with clear inputs/outputs.
3. **Deterministic where possible, LLM where necessary.** Pattern matching is deterministic (BaseAgent). Anomaly analysis, instruction generation, investigation, and synthesis use LLMs.
4. **Skip logic is minimal and deterministic.** Only one skip: Phase 3 is skipped when Phase 2 returns high-confidence. All other phases are mandatory.
5. **Tools are organized by purpose.** 14 tool files in `tools/`, each with a single clear responsibility. Network diagnostic tools and ontology query tools coexist as tools for understanding the network.
6. **The ontology is the network's knowledge.** It's not a separate system to "bridge" to — it's a formal representation of the network itself, accessible via tools like any other aspect of the network.
